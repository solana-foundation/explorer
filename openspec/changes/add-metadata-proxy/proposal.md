# Proposal: Server-side proxy for off-chain metadata and images

## Context

- The Explorer renders token/NFT artwork, logos, and off-chain JSON metadata from URIs that originate **on-chain** — i.e. supplied by arbitrary third parties (token creators, NFT mints, `security.txt` logos). Today the browser fetches those URIs **directly**: `getProxiedUri` (`app/features/metadata/utils.ts`) returns the raw URI, and components like `NFTArt`, `AccountHeader`, `OwnedTokensCard`, `CompressedNftCard`, and the receipt page load it straight from the upstream host.
- Direct browser fetching has four standing problems for user-supplied URIs:
  - **Privacy** — every viewer's IP and `Referer` are exposed to whatever host a token creator put on-chain.
  - **CORS reliability** — many metadata hosts (S3, IPFS gateways, game CDNs) don't send `Access-Control-Allow-Origin`, so JSON fetches fail in the browser even though the bytes are public.
  - **No resource control** — the browser will pull a multi-hundred-MB "image" with no size or time bound.
  - **Untrusted content execution** — an SVG with embedded `<script>` served from a metadata host runs in the Explorer origin if rendered or opened directly.
- A server-side proxy route already exists under `app/api/metadata/proxy/` but is **gated off**: the route returns `404` unless `NEXT_PUBLIC_METADATA_ENABLED === 'true'`, and `getProxiedUri` only rewrites URIs to `/api/metadata/proxy?uri=…` when that same flag is set. The feature is being built and validated dark; this proposal is the design record for what the proxy guarantees and why, ahead of enabling it per-deployment.

## Why

We want a **single, SSRF-safe server seam** through which all off-chain metadata and image bytes flow, so we can apply privacy, reliability, resource, and content-safety policy in one place instead of trusting the browser to fetch arbitrary on-chain URIs.

Principles that drove the design:

- **A proxy is the only seam that owns the fetch.** Privacy (hide the viewer's IP), CORS reliability (server-to-server has no CORS), size/timeout limits, and content sandboxing all require the fetch to happen on our side. No client-only change achieves them.
- **User-supplied URIs are hostile until proven otherwise.** The target host comes from on-chain data, so the proxy must assume SSRF intent: it validates every hop against private IP ranges, pins DNS to defeat rebinding, and refuses non-HTTP protocols.
- **Expected-but-unwanted input is not a fault.** Oversize bodies, missing CORS, and unsupported content types are normal third-party behaviour. They produce deterministic HTTP statuses and (for the size cap and network failures) `warning`-level Sentry signal — not exceptions that page on-call.
- **Cap near the platform ceiling, degrade gracefully.** Rather than raising the cap to fit every large image (which Vercel's ~4.5 MB buffered-response limit blocks anyway), keep the cap just under that limit (4 MB) to bound the memory the `force-dynamic` function buffers, and handle the rejected tail in the UI: oversize/failed images render a fallback that names the failure and links to the original instead of a broken-image icon. The cap is chosen from real traffic — the success path logs fetched sizes so the distribution can be queried before settling the number.
- **Ship dark, enable deliberately.** Turning the proxy on reroutes every metadata/image request through one function and changes the failure surface; it stays behind `NEXT_PUBLIC_METADATA_ENABLED` so it can be validated on preview and enabled per-deployment, with the limit tunable via env before launch.

Alternatives considered:

- **Keep direct browser fetching (the current disabled-state behaviour).** Zero infra. Rejected as the long-term posture: it cannot fix the privacy leak, the CORS failures, or untrusted-SVG execution, and gives no resource bound — exactly the four problems above.
- **Use a third-party image proxy / CDN (imgix, weserv, Cloudflare Images).** Offloads bytes and resizing. Rejected: it adds a vendor dependency and cost, still requires an SSRF-safe fetch of arbitrary on-chain URIs (the CDN would fetch whatever we point it at), and gives us no control over the JSON path or the content-type/sandbox policy.
- **Use Next.js built-in image optimization (`next/image` loader).** Native. Rejected: it's designed for a known allow-list of image domains, not arbitrary user-supplied hosts, and does not cover the JSON-metadata path at all.
- **Validate only the initial hostname, then let `fetch` follow redirects.** Simplest proxy. Rejected: it reopens the SSRF hole — a public host can `302` to `169.254.169.254` (cloud metadata) or an internal address. The proxy instead follows redirects manually with per-hop re-validation.
- **Trust `Content-Length` alone for the size cap.** Cheap. Rejected: a hostile or sloppy upstream can omit or understate it, so the cap is also enforced on the actual streamed byte count and on the decode path.
- **Raise the cap to fit large images (e.g. 10 MB).** Tempting — it would proxy more images successfully. Rejected: Vercel rejects buffered responses above ~4.5 MB regardless, so a 10 MB cap mostly buffers bytes the platform will refuse anyway, costing memory on a `force-dynamic` route that buffers the whole body. A 4 MB cap (just under the platform ceiling) plus a graceful client fallback covers the tail without paying for it.
- **Show one generic fallback with no reason (zero extra requests).** Cheapest: a single on-`error` fallback covers every cause (413/404/502/CORS) and adds no request. Initially chosen, now rejected: a viewer can't tell an oversize image from a missing one or an unsupported type, so the fallback can't guide them ("View original" is futile for a 404). The cost objection that drove the original choice is removed by briefly browser-caching the error response (`private, max-age=30`): the failed `<img>` request primes the cache, so the on-error reason probe re-reads the status from cache instead of re-invoking the proxy. The probe re-`fetch`es the same-origin proxy URL (not the upstream), so it adds no privacy leak. The 413 copy stays generic ("maximum size") rather than naming the exact cap — least-disclosure, and correct if the cap is tuned.
- **Block on size data before choosing the cap.** Safest. Rejected: we set 4 MB now (just under Vercel's limit) and instrument the success path concurrently, then tune from the observed distribution rather than gating the rollout on data collection.
- **Enable the proxy by default once merged.** Rejected: enabling reroutes all metadata/image traffic through one function and changes the failure modes site-wide; a flag lets us validate on preview and roll out per-deployment.

## What Changes

The metadata proxy comprises a Next.js route, a `feature/` library of SSRF-safe fetch primitives, and a client URI rewriter, all gated by one flag:

- **Route — `app/api/metadata/proxy/route.ts`.** A `force-dynamic` `GET` that returns `404` when `NEXT_PUBLIC_METADATA_ENABLED !== 'true'`, parses the `uri` query param, rejects non-HTTP protocols, and delegates to `fetchResource`. It maps any thrown `StatusError` to a canonical status body (`STATUS_MESSAGES`), attaches hardening headers, strips `Content-Length` so the response stays CORS-simple, forwards `content-type`, and sets a browser-only `Cache-Control` (see **Browser caching** below) rather than forwarding the upstream `cache-control`. It does **not** forward the upstream `etag` — the route does no conditional revalidation, so a forwarded validator would be inert.
- **SSRF-safe fetch — `feature/fetch-resource.ts` + `feature/ip.ts`.** Each hop resolves DNS once, validates every returned address against private IPv4/IPv6 ranges (and localhost), and replays those pinned addresses to undici so the kernel never re-resolves — closing the DNS-rebinding TOCTOU window. Redirects are followed manually up to `MAX_REDIRECTS` (3) with per-hop re-validation and loop detection.
- **Resource limits — `feature/read-body-with-limit.ts` + `config.ts`.** A per-request byte cap (default 4 MB, override `NEXT_PUBLIC_METADATA_MAX_CONTENT_SIZE`) is enforced at the `Content-Length` pre-check, mid-stream, on `fetch()` rejection, and at decode. A request timeout (default 10 s, override `NEXT_PUBLIC_METADATA_TIMEOUT`) bounds each hop, and `export const maxDuration = 15` caps the whole invocation at the Vercel platform layer — covering DNS resolution and multi-hop redirect chains that the per-hop timeout does not bound.
- **Browser caching — `route.ts`.** Successful (2xx) responses carry a browser-facing `Cache-Control: public, max-age=86400` (1 day), so a viewer's own browser serves repeat views from its cache for up to a day without re-hitting the function. A day-long cache is safe because the proxied content is effectively immutable (content-addressed Arweave/IPFS, static CDN assets); `immutable` is intentionally **not** set, so a rare in-place overwrite of a mutable URL is still picked up once the window lapses. Caching is browser-only by design: no `Vercel-CDN-Cache-Control` is set, so responses are **not** cached at the Vercel edge and every cold or cross-client view re-invokes the function. Errors carry a short browser-only `Cache-Control: private, max-age=30` (never edge-cached) so a failed `<img>` request primes the cache for `ProxiedImage`'s on-error reason probe — see the client-fallback bullet — while a transient `5xx` still clears quickly. The upstream's own `Cache-Control` is deliberately **not** forwarded — it is frequently `no-store`/`no-cache`/`private`.
- **Content policy — `feature/processors.ts`.** Only `application/json`, `text/plain` (parsed as JSON), and `image/*` are served; anything else is `415`. Malformed JSON is `415`, not `500`.
- **Response hardening — `route.ts`.** A `Content-Security-Policy: sandbox; default-src 'none'; …` plus `X-Content-Type-Options: nosniff` prevents proxied SVG/HTML from executing if the proxy URL is opened directly.
- **Observability.** The size-cap paths and the network-failure path emit `Logger.warn(..., { sentry: true, sentryExtras })` with `host`/`maxSize` context; the **success path** emits a `Logger.info('[api:metadata-proxy] Resource fetched', { byteLength, contentType, host, maxSize })` so the full fetched-size distribution (not just the over-cap tail) can be queried from logs to tune the cap; SSRF blocks, redirect anomalies, and non-2xx upstreams are logged.
- **Client rendering & fallback — `app/features/metadata/ui/ProxiedImage.tsx`.** A `ProxiedImage` component owns the full lifecycle of every off-chain image: a slot-sized skeleton while loading, the image on success, and a graceful fallback on any load failure. It composes proxy-agnostic primitives from `app/components/shared/ui/image/`: `ImageWithFallback` (an `<img>` that shows a `placeholder` while loading and swaps in a `fallback` on a missing/failed `src`) and `ExternalResourceLink` (the opt-in "View original" affordance). Because an `<img>` can't read the HTTP status, on failure the component re-`fetch`es the same-origin proxy `src` — via `model/useImageFailureReason` → `lib/imageFailure` — to read the status and map it to a per-status reason (413 oversize, 404, 415, 502, 504); the failed `<img>` request primed the browser cache, so the probe is a cache hit rather than a fresh round-trip, and it hits the proxy (not the upstream), adding no privacy leak. The default fallback is a Solana-logo placeholder that inherits the image's own className/box (a rounded-full avatar gets a round logo) and carries the reason as its accessible name and a hover tooltip; consumers can pass a `fallback` render function to receive the reason instead. `ProxiedImage` is the single rendering path for every proxied image: the prominent NFT displays (`NFTArt` → `MetaplexNFTHeader`/`CompressedNftCard`, `NFTokenAccountSection`) opt into the "View original" link via `showOriginalLink` — where `ExternalResourceLink`'s `detail` prop surfaces the reason in the link's info tooltip — while the small logo slots (`AccountHeader` token logo, `OwnedTokensCard` row icon, `ProgramHeader` logo) render the default fallback without the link, so a failed or oversize load degrades to the slot-shaped logo (with the reason on hover) instead of a broken-image icon.
- **Client gate — `app/features/metadata/utils.ts`.** `getProxiedUri` rewrites `http(s)` URIs to the proxy path when the flag is on and passes everything through unchanged when off. `ProxiedImage` calls it internally, so image render sites pass a raw `uri` and never touch the gate directly; the remaining direct callers are the JSON/metadata fetch paths (`useOffChainMetadata`, the `AccountHeader`/`CompressedNftCard` metadata links, `MetaplexNFTAttributesCard`, `entities/nft`, the receipt page). Either way, flipping the flag needs no per-call-site change.

## Impact

- **Runtime surface.** With the flag **off** (current default), behaviour is unchanged — URIs are fetched directly by the browser and the route 404s. With the flag **on**, every metadata/image request routes through `/api/metadata/proxy`, gaining the privacy/CORS/size/sandbox guarantees and the proxy's failure modes (`403` SSRF block, `413` oversize, `415` unsupported type, `502` bad gateway, `504` timeout); images that the proxy rejects render the `ProxiedImage` fallback rather than a broken-image icon.
- **Configuration.** `NEXT_PUBLIC_METADATA_ENABLED` (master switch), `NEXT_PUBLIC_METADATA_MAX_CONTENT_SIZE` (cap, default 4 MB), `NEXT_PUBLIC_METADATA_TIMEOUT` (default 10 s), `NEXT_PUBLIC_METADATA_USER_AGENT` (default `Solana Explorer`).
- **Cap tuning (open follow-up).** 4 MB is the initial value; the success-path size log feeds the decision on the long-term cap (and a possible split between a small JSON cap and a larger image cap). Lowering the cap increases how often the `ProxiedImage` fallback shows, so the chosen number trades fallback frequency against memory/egress.
- **Accepted risk (Vercel buffered-response cap).** Vercel Functions cap buffered response bodies at ~4.5 MB (`FUNCTION_RESPONSE_PAYLOAD_TOO_LARGE`). The 4 MB default sits just below that — a thin ~0.5 MB margin — so our own limit remains the binding constraint and oversize content is handled by the client fallback rather than a platform-layer rejection, but the headroom is small: if real payloads (body plus any response overhead) crowd the 4.5 MB line, the cap should be trimmed back. The success-path size log is what tells us whether that's happening.
- **Accepted risk (centralized load & egress).** Enabling concentrates all metadata/image fetching on one function and shifts egress from the viewer to our infrastructure. A day-long browser `Cache-Control` (`public, max-age=86400`) lets a viewer's own browser serve repeat views without re-hitting the function, but responses are **not** edge-cached (no `Vercel-CDN-Cache-Control`), so cold and cross-client views still invoke the function; this residual concentration is the deliberate trade for the privacy and safety guarantees, and is why the cap is bounded just under the platform ceiling and the rollout flag-gated.
- **Tests.** `app/api/metadata/proxy/__tests__/` covers the SSRF/private-IP matrix, redirect handling, size-cap paths (incl. Sentry assertions), content-type routing, and the route endpoint (incl. the short error `Cache-Control`); an integration spec exercises the real undici streaming path. The failure-reason layer has unit tests — `lib/__tests__/imageFailure.spec.ts` (status→reason map, probe caching, unreadable-status fallback) and `model/__tests__/useImageFailureReason.spec.ts` (on-error probe, reset on src change, no probe off-origin) — and `ImageWithFallback` is unit-tested for the placeholder swap and the cached-image event recovery. The end-to-end client states are covered by Storybook stories with `play` assertions — `app/features/metadata/ui/__stories__/ProxiedImage.stories.tsx` (loaded, with-link, generic fallback, oversize-413 reason, fallback-with-link, loading skeleton, and no-image states) — built on the shared primitives' own stories under `app/components/shared/ui/image/stories/`.
